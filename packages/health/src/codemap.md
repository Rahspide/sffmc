## Responsibility

Source directory for @sffmc/health. Contains the complete implementation: all check logic, the orchestrator, the OpenCode plugin registration wrapper, and its test suite.

## Design

- **Single-file module** — no internal sub-modules. All logic in `index.ts`. This is intentional: the plugin is a bounded, self-contained diagnostic tool, not a framework.
- **CheckFn type alias** — `(repoRoot: string) => Promise<CheckResult>`. Every check is a pure function of the repo root. No shared mutable state, no class instances, no constructor injection.
- **Status enum** — `"ok" | "warn" | "fail"` (not a TypeScript enum). `warn` is advisory (version mismatches, missing license refs); `fail` is blocking (missing audit script, missing tests, type errors, tool registration bugs, missing LICENSE).
- **`packageNames()` helper** — discovers all packages in the monorepo. Iterates `packages/` directories plus conditionally adds `shared/` if it has a `package.json`. This is the single source of truth for "what is a package" across all 7 checks.
- **Indent-aware parser in `checkToolRegistration`** — walks source lines tracking indent depth to distinguish tool-level `name: "string"` from parameter-schema `name: { type: "string" }`. Uses `keysByIndent` + `stringKeysByIndent` Maps. Skips comments (`//`, `/*`-blocks).
- **`checkHookConflicts` safelist** — 14 hooks classified safe-for-multi-plugin. Design rationale: most OpenCode hooks chain/aggregate; only structural-exclusive hooks (like `server` or `workspace`) would conflict. The safelist filters out benign overlaps from the audit report.
- **`Bun.spawn` for external commands** — uses native Bun subprocess API (not `child_process`), pipes both stdout and stderr, awaits `proc.exited`. Consistent pattern across `checkHookConflicts` (python3) and `checkTypeCheck` (bun build).

## Flow

```
Plugin loaded by OpenCode
  → server(ctx) called
    → extracts ctx.projectRoot as repoRoot
    → returns { tool: { sffmc_health: { description, parameters, execute } } }
  ── LLM invokes sffmc_health() ──
    → execute() called
      → runAllChecks(repoRoot)
        → Promise.all(ALL_CHECKS.map(fn => fn(repoRoot)))
          [7 parallel checks, each self-contained]
        → aggregate counts by status
        → return { ok, checks, summary }
      → JSON.stringify(result, null, 2)
    → returned to LLM as string
```

Each check internally:
1. Discover packages via `packageNames(repoRoot)`
2. For each package, apply package-specific logic (read fs, spawn command, parse output)
3. Collect missing/failed items in an array
4. Return `CheckResult` with appropriate status and detail

## Integration

- **Import from `@sffmc/shared`** — `PluginContext` type only (used in `server` signature). No runtime dependency on shared logic.
- **Filesystem access via `node:fs/promises`** — `readdir`, `readFile`, `stat` for all check I/O. No sync fs calls.
- **Path construction via `node:path`** — `join` for all `repoRoot`-relative path assembly. No hardcoded separators.
- **External subprocess spawns** — `Bun.spawn(["python3", scriptPath])` for audit script, `Bun.spawn(["bun", "build", ...])` for typecheck. Both expect the relevant executables (`python3`, `bun`) on PATH.
- **External files read** — `.sffmc/load-order-audit.json` (generated by audit script), `packages/*/src/index.ts` (typecheck entry), `packages/compose/src/index.ts` and `packages/workflow/src/tool.ts` (tool registration check).
